VLANs avançades, Router-on-a-Stick i ACLs
M08 5 sessions · ~10h Prerequisit: P1 completat
S1 — VLANs per departament i disseny de l'adreçament
›Revisió del pla d'adreçament del P1 i extensió per a 5 VLANs de producció + VLAN de gestió (99) + VLAN nativa (999).
›Creació de VLANs al switch: comandes vlan [id], name, assignació de ports d'accés.
›Configuració del port trunk entre switch i router: switchport mode trunk, gestió de VLANs permeses, VLAN nativa.
›Verificació amb show vlan brief, show interfaces trunk.
S2–S3 — Router-on-a-stick complet i Inter-VLAN routing
›Subinterfícies Cisco IOS: encapsulation dot1Q [vlan-id], assignació d'IPs (gateways de cada VLAN).
›Flux d'un paquet inter-VLAN pas a pas: PC → switch (tag) → router (desencapsulat, decision de routing, re-encapsulat) → switch → PC destí.
›Diagnòstic: show ip route, show ip interface brief, ping des del router a cada gateway.
S4–S5 — ACLs: matriu d'accessos entre VLANs
›Repàs d'ACLs estàndard vs exteses. Regla implicit deny. Processament seqüencial.
›Disseny de la matriu d'accessos: IT accedeix a tot; Direcció accedeix a Vendes i RRHH; Logística i Vendes entre elles; RRHH no accedeix a Logística ni Vendes.
›Implementació: ACLs exteses named, aplicació a les subinterfícies del router (inbound), verificació i troubleshooting.
›Regles especials: permetre DHCP (UDP 67/68) i DNS (UDP/TCP 53) independentment de la matriu d'accessos.

Active Directory, LDAP, GPOs i scripting PowerShell
M06 6 sessions · ~12h
S1–S2 — LDAP i Active Directory: arquitectura i conceptes
›Necessitat d'un directori centralitzat: problema de les N×M credencials (N usuaris × M serveis). El directori resol: un compte → accés a tots els serveis autoritzats.
›LDAP (Lightweight Directory Access Protocol): protocol estàndard per consultar i modificar directoris. Estructura jeràrquica (DIT - Directory Information Tree). DN (Distinguished Name): CN=Joan Garcia,OU=IT,DC=netcorp,DC=local.
›Active Directory Domain Services (AD DS): implementació Microsoft d'un directori basat en LDAP + Kerberos. Conceptes: domini, forest, arbre, controlador de domini (DC), catàleg global.
›Objectes AD: Usuaris (User), Grups (Group), Equips (Computer), Unitats organitzatives (OU). Atributs: sAMAccountName, userPrincipalName, memberOf, distinguishedName.
S3 — Instal·lació AD DS i estructura d'OUs
›Instal·lació del rol AD DS a Windows Server: Server Manager → Add Roles → Promote to Domain Controller. Elecció del nom de domini intern (netcorp.local).
›Disseny de l'estructura d'OUs per a NetCorp: OU per departament (Direccio, IT, Logistica, Vendes, RRHH) + OUs funcionals (Servidors, Grups, GPOs). Criteris de disseny: per geografia, per departament o per funció.
›Grups de seguretat vs grups de distribució. Àmbit dels grups: Domain Local, Global, Universal. Bones pràctiques: estratègia AGDLP (Account → Global → Domain Local → Permission).
S4 — Group Policy Objects (GPO)
›Què és una GPO: conjunt de configuracions que s'apliquen automàticament als usuaris i equips d'una OU quan inician sessió o arrenquen. Evita la configuració manual de cada PC.
›Aplicació de GPOs: vinculació a Site, Domini o OU. Herència i precedència (LSDOU). Filtratge de seguretat. Forçar actualització: gpupdate /force.
›Exemples de GPOs útils per a NetCorp: mapeig de unitats de xarxa per departament, restricció del panell de control, política de contrasenyes, bloqueig de USB, fons d'escriptori corporatiu, configuració del servidor DNS als clients.
S5–S6 — PowerShell i scripting per a gestió massiva
›PowerShell per a AD: mòdul ActiveDirectory. Cmdlets fonamentals: New-ADUser, Get-ADUser, Set-ADUser, Add-ADGroupMember.
›Lectura de fitxers CSV amb Import-Csv. Bucle ForEach-Object per processar cada fila. Generació automàtica de contrasenyes i noms de login (nom.cognom).
›Gestió d'errors en PowerShell: Try/Catch, -ErrorAction, log de resultats a fitxer. Script idempotent: si l'usuari ja existeix, no falla.
›Alternativa bash + ldap-utils per a entorns Linux (OpenLDAP): ldapadd, ldapmodify, fitxers LDIF.

DNS: zones, registres i resolució de noms
M07 4 sessions · ~8h
S1 — Com funciona DNS: el sistema de noms d'Internet
›El problema: els humans recorden noms (google.com), els ordinadors treballen amb IPs (142.250.185.46). DNS (Domain Name System, RFC 1034/1035) és el sistema que tradueix entre ells.
›Jerarquia DNS: arrel (.) → TLD (.com, .cat, .org) → domini de segon nivell (google.com) → subdominis (mail.google.com). Cada nivell té servidors autoritaris responsables.
›Resolució recursiva vs iterativa: el resolver del client demana al servidor DNS local (recursiu), que fa les consultes iteratives fins als servidors autoritaris i retorna la resposta. Caché: les respostes es guarden el temps indicat pel TTL.
›DNS per a xarxes internes: el domini netcorp.local és un domini privat que no existeix a Internet. El servidor DNS intern el resol per als clients de la xarxa local.
S2–S3 — Zones DNS i tipus de registres
›Zona directa (Forward zone): resol noms → IPs. Ex: dc.netcorp.local → 192.168.20.10.
›Zona inversa (Reverse zone): resol IPs → noms (PTR records). Ex: 10.20.168.192.in-addr.arpa → dc.netcorp.local. Necessari per a molts serveis (correu, SSH, autenticació).
›Registres DNS fonamentals: A (nom → IPv4), AAAA (nom → IPv6), CNAME (àlies → nom canònic), MX (servidor de correu del domini), PTR (zona inversa), NS (servidors de noms de la zona), SOA (inici d'autoritat, paràmetres de la zona), TXT (text lliure; usat per SPF, DKIM).
›TTL (Time To Live): temps en segones que un registre es pot guardar en caché. TTL baix = canvis ràpids; TTL alt = menys càrrega als servidors. Per a servidors interns: 3600 s és habitual.
›Zones primàries vs secundàries: la primària té l'arxiu de zona en lectura/escriptura. La secundària és una còpia de seguretat, actualitzada mitjançant transferències de zona (AXFR, IXFR).
S4 — BIND9 a Linux i DNS dinàmic
›BIND9 (Berkeley Internet Name Domain): el servidor DNS de codi lliure de referència. Fitxers de configuració: named.conf (configuració principal), named.conf.local (zones), fitxers de zona (format RFC 1035).
›DNS dinàmic (DDNS): el servidor DHCP actualitza automàticament el DNS quan assigna una IP. Integra DHCP i DNS perquè els noms dels clients sempre resolvin correctament sense configuració manual.
›Diagnòstic DNS: dig (consultes detallades), nslookup (Windows i Linux), host, named-checkzone (validar fitxer de zona).

Amenaces i assegurament del servei DNS
C037 3 sessions · ~6h
S1 — Atacs al DNS: com funcionen i per què són perillosos
›DNS Cache Poisoning: l'atacant aconsegueix que el servidor DNS guardi en caché una resposta falsa (ex: google.com → IP de l'atacant). Tots els clients que consultin aquell servidor seran redirigits al lloc fals sense saber-ho. Atac de Kaminsky (2008): explotava la previsibilitat del port UDP i l'ID de transacció.
›DNS Amplification (DDoS): l'atacant fa consultes DNS amb IP origen falsificada (la IP víctima). El servidor DNS respon amb paquets molt més grans a la víctima. Amplificació: una consulta de 40 bytes pot generar una resposta de 4000 bytes (factor ×100). Usat en atacs DDoS massius.
›DNS Hijacking: modificació dels registres DNS legítims al servidor autoritari (per accés no autoritzat) per redirigir el trànsit. Diferent del cache poisoning: afecta el servidor original, no la caché.
›DNS Tunneling: ús del protocol DNS com a canal de comunicació encobert. Les dades s'encapsulen en consultes DNS (subdominins llargs). Evasió de firewalls que permeten DNS però bloquegen altres protocols.
S2 — Configuració segura de BIND9 i Windows DNS
›Recursivitat restringida: configurar el servidor DNS intern perquè només faci resolució recursiva per als clients de la xarxa local, no per a qualsevol IP de l'exterior (evita ser usat com a amplificador DDoS).
›Transferències de zona restringides (allow-transfer): permetre les transferències de zona (AXFR) únicament a les IPs dels servidors DNS secundaris autoritzats. Evita que qualsevol entitat descarregui tots els registres del domini intern.
›Randomització del port origen (port randomization): cada consulta DNS usa un port UDP origen aleatori, dificultant els atacs de cache poisoning basats en predir el port.
›Separació DNS intern / extern (split-horizon DNS): el servidor DNS intern resol les IPs privades per als clients interns, però mai exposa els registres interns a l'exterior. El servidor DNS públic (si n'hi ha) publica únicament els serveis públics.
S3 — DNSSEC: autenticació dels registres DNS
›Problema que resol DNSSEC: DNS original no té autenticació. Un client no pot saber si la resposta DNS que rep és legítima o ha estat manipulada. DNSSEC afegeix signatura digital als registres DNS.
›Funcionament: cada zona té un parell de claus (KSK - Key Signing Key i ZSK - Zone Signing Key). Els registres DNS es signen digitalment amb la ZSK. El client pot verificar la signatura i detectar si la resposta ha estat manipulada.
›Registres DNSSEC nous: RRSIG (signatura del registre), DNSKEY (clau pública de la zona), DS (delegació de signatura, a la zona pare), NSEC/NSEC3 (prova de no existència autenticada).
›Limitacions pràctiques: DNSSEC no xifra les respostes (continuen sent llegibles), no protegeix contra DDoS. Increment de la mida de les respostes. Complexitat de gestió de claus i renovació.
