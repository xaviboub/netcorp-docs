Encaminament dinàmic (RIP/OSPF) i NAT avançat
M08 5 sessions · ~10h Prerequisit: P1+P2 (router-on-a-stick i ACLs)
S1 — Per què calen protocols d'encaminament dinàmic
›Límits de les rutes estàtiques: cada nova xarxa (com el túnel cap a Azure) requereix configuració manual a TOTS els routers. En xarxes amb múltiples seus, això és inviable i propens a errors.
›Protocols de vector de distància vs estat d'enllaç: RIP (vector de distància, mètrica = nombre de salts, màx. 15) vs OSPF (estat d'enllaç, mètrica = cost basat en ample de banda, sense límit de salts).
›Distància administrativa: com decideix el router quina ruta usar quan coneix la mateixa xarxa per diversos mitjans (directa=0, estàtica=1, OSPF=110, RIP=120).
S2 — RIPv1 i RIPv2: configuració i limitacions
›Configuració bàsica: router rip, version 2, network [xarxa] per cada xarxa directament connectada a anunciar.
›RIPv1 (classful, sense màscara, broadcast) vs RIPv2 (classless, suporta VLSM, multicast 224.0.0.9). Per què RIPv1 no funciona amb el VLSM del Projecte 1.
›Limitacions de RIP: convergència lenta (timers de 30s, fins a 180s per detectar caiguda), límit de 15 salts, mètrica simplista (no considera ample de banda).
›Diagnòstic: show ip route (rutes marcades amb R), show ip protocols, debug ip rip.
S3–S4 — OSPF (Open Shortest Path First)
›Conceptes fonamentals: àrees OSPF (Area 0 = backbone), adjacències (relacions Router ↔ Router via Hello packets), algorisme de Dijkstra per calcular el camí més curt (SPF tree).
›Configuració: router ospf [process-id], network [xarxa] [wildcard] area 0. Wildcard mask: invers de la màscara de subxarxa.
›Router ID: identificador únic de 32 bits (format IP) que identifica el router dins OSPF. S'elegeix automàticament (IP loopback més alta, o IP d'interfície més alta si no hi ha loopback) o manualment.
›Cost OSPF: inversament proporcional a l'ample de banda (cost = 10^8 / BW). Influeix en la selecció de la ruta òptima quan hi ha múltiples camins.
›Diagnòstic OSPF: show ip ospf neighbor (estat FULL = adjacència establerta), show ip ospf database, show ip route (rutes marcades amb O).
S5 — NAT avançat: publicar serveis a Internet
›Repàs PAT/NAT Overload (vist al P1) i introducció al NAT estàtic 1:1 per a servidors que han de ser accessibles des de fora amb IP fixa.
›Port Forwarding (NAT estàtic amb traducció de port): publicar un servei intern (ex: 192.168.40.10:80) a través d'un port concret de la IP pública del router. Sintaxi Cisco: ip nat inside source static tcp [IP_privada] [port] [IP_pública] [port] extendable.
›Implicacions de seguretat: cada regla NAT estàtica obre un forat al firewall implícit del NAT. Cal combinar-ho amb ACLs per limitar l'origen del trànsit permès.
›Verificació: show ip nat translations, show ip nat statistics, prova des d'un dispositiu extern (mòbil amb dades, no WiFi de l'aula).

Servidors web, FTP i correu electrònic
M07 8 sessions · ~16h
S1–S2 — Servidors web: Apache i Nginx
›Funcionament d'un servidor web: rep peticions HTTP, localitza el recurs (fitxer estàtic o executa un programa), retorna una resposta HTTP amb codi d'estat (200, 404, 500...).
›Apache (httpd): arquitectura modular (mod_ssl, mod_rewrite, mod_security). Fitxers de configuració: /etc/apache2/sites-available/, a2ensite, a2enmod.
›Nginx: arquitectura basada en esdeveniments (event-driven), més eficient per a alta concurrència. Sovint usat com a proxy invers davant d'Apache.
›VirtualHosts (Apache) / Server Blocks (Nginx): permeten servir múltiples llocs web des d'un mateix servidor, diferenciats per nom de domini (Name-based) o per IP/port (IP-based).
S3 — SSL/TLS i certificats digitals
›Per què HTTPS: xifra la comunicació client-servidor (confidencialitat), verifica la identitat del servidor (autenticitat) i garanteix que les dades no s'han modificat (integritat).
›Certificat digital: conté la clau pública del servidor i està signat per una Autoritat de Certificació (CA). El navegador confia en CAs preinstal·lades.
›Let's Encrypt i Certbot: CA gratuïta i automatitzada. Per a dominis públics (la web corporativa publicada). Per a serveis interns (intranet .local), s'usen certificats autosignats o una CA pròpia.
›Redirecció HTTP→HTTPS i HSTS (HTTP Strict Transport Security): força el navegador a usar només HTTPS per a aquest domini.
S4–S5 — Servei FTP: vsftpd
›FTP usa DOS connexions: control (port 21, comandes) i dades (port 20 en mode actiu, port aleatori en mode passiu). Aquesta dualitat és el motiu pel qual FTP té problemes amb NAT i firewalls.
›Mode actiu: el SERVIDOR inicia la connexió de dades cap al client (problema: el client sol estar darrere NAT, el servidor no pot connectar-hi). Mode passiu: el CLIENT inicia ambdues connexions (recomanat darrere NAT).
›vsftpd (Very Secure FTP daemon): configuració a /etc/vsftpd.conf. Paràmetres clau: pasv_enable, pasv_min_port/pasv_max_port (rang de ports per al mode passiu, cal obrir-los al firewall).
›Usuaris virtuals (no comptes de sistema reals): cada departament té un usuari FTP amb el seu directori chroot (chroot_local_user=YES), sense accés SSH ni shell al sistema.
›FTP vs SFTP vs FTPS: FTP transmet credencials en text pla (insegur). SFTP (sobre SSH, port 22) i FTPS (FTP + TLS) són les alternatives segures — es tractaran a la fase de seguretat.
S6–S8 — Correu electrònic: Postfix, Dovecot i antispam
›Arquitectura del correu: MTA (Mail Transfer Agent, ex: Postfix) envia/rep correu entre servidors via SMTP (port 25/587). MDA/MAA (Mail Delivery/Access Agent, ex: Dovecot) permet als usuaris accedir a la seva bústia via IMAP (port 143/993) o POP3 (port 110/995).
›SMTP: protocol per ENVIAR correu (de client a servidor, o de servidor a servidor). El registre MX del DNS indica quin servidor rep el correu d'un domini.
›IMAP vs POP3: IMAP sincronitza l'estat dels correus amb el servidor (llegit/no llegit, carpetes) — ideal per a múltiples dispositius. POP3 descarrega i normalment elimina els correus del servidor.
›Postfix: configuració a /etc/postfix/main.cf. Paràmetres clau: myhostname, mydomain, mynetworks (xarxes autoritzades a enviar sense autenticació).
›Dovecot: servidor IMAP/POP3. Integració amb Postfix mitjançant un mateix magatzem d'usuaris (ex: usuaris del sistema o base de dades).
›SpamAssassin: analitza cada correu entrant amb un sistema de puntuació basat en regles (capçaleres sospitoses, paraules clau, llistes negres DNSBL). Per sobre d'un llindar, el correu es marca com a spam o es rebutja.
›Open relay: un servidor de correu mal configurat que permet a qualsevol enviar correu a través d'ell. És un dels errors de configuració més greus — convertiria NetCorp en font de spam mundial. mynetworks ha d'estar ben restringit.

Accés remot segur i integració de sistemes
M06 4 sessions · ~8h
S1–S2 — SSH avançat: tunneling i RDP/VNC
›Repàs d'autenticació per clau pública (vista al P1) aplicada ara a TOTS els nous servidors (web, FTP, correu).
›SSH Tunneling (port forwarding): ssh -L [port_local]:[host]:[port_remot] usuari@servidor (local forwarding, accedir a un servei intern a través del servidor SSH). ssh -D [port] (proxy SOCKS dinàmic).
›Cas d'ús a NetCorp: un comercial fora de l'oficina vol accedir a la intranet (servidor web intern, no publicat). Amb un túnel SSH cap al servidor, pot accedir-hi sense exposar la intranet a Internet.
›RDP (Remote Desktop Protocol, port 3389): accés gràfic remot a Windows. VNC (Virtual Network Computing): alternativa multiplataforma per a Linux/macOS/Windows. Tots dos s'han de tunelitzar via SSH o VPN, mai exposats directament a Internet.
S3–S4 — Samba: integració Windows-Linux i comparticions
›Samba implementa el protocol SMB/CIFS de Microsoft sobre Linux. Permet que un servidor Linux comparteixi carpetes que els PCs Windows veuen com a unitats de xarxa normals.
›Integració amb Active Directory: el servidor Samba s'uneix al domini netcorp.local (com a "Domain Member"), de manera que els permisos de les comparticions es basen en els usuaris i grups d'AD ja existents (els del P2).
›Configuració a /etc/samba/smb.conf: secció [global] (configuració del domini) i una secció per cada compartició, amb permisos basats en grups AD (valid users = @"NETCORP\\GRP_Logistica").
›NFS (Network File System): alternativa per a comparticions entre servidors Linux (no requereix autenticació Windows). Útil per a comparticions entre les VMs de Proxmox i la VM Azure.

Seguretat web: OWASP Top 10
C037 4 sessions · ~8h
S1 — OWASP Top 10: panoràmica i injeccions
›OWASP (Open Worldwide Application Security Project): organització que manté la llista de les 10 vulnerabilitats web més crítiques i freqüents, actualitzada periòdicament. Referència estàndard del sector.
›Injecció SQL: quan una aplicació web incorpora directament l'entrada de l'usuari a una consulta SQL sense validar-la. Permet a un atacant llegir, modificar o eliminar dades de la base de dades. Mitigació: consultes parametritzades (prepared statements), mai concatenació de strings.
›XSS (Cross-Site Scripting): injecció de codi JavaScript que s'executa al navegador d'altres usuaris. XSS reflectit (via URL/paràmetres), XSS emmagatzemat (guardat a la BD, ex: en comentaris). Mitigació: escapar/sanititzar tota sortida HTML.
›CSRF (Cross-Site Request Forgery): un lloc maliciós fa que el navegador de la víctima (que té sessió oberta en un altre lloc) executi accions no desitjades en aquell lloc. Mitigació: tokens CSRF únics per formulari.
S2 — Gestió insegura de sessions i autenticació
›Cookies de sessió: han d'incloure els atributs Secure (només per HTTPS), HttpOnly (no accessible per JavaScript, mitiga XSS) i SameSite (mitiga CSRF).
›Gestió de contrasenyes: mai en text pla a la BD. Hash amb sal (bcrypt, Argon2). Política de complexitat i bloqueig per intents fallits.
›Configuració errònia de seguretat (Security Misconfiguration): un dels riscos més comuns. Exemples: pàgines d'administració accessibles sense autenticació, missatges d'error que revelen informació interna (stack traces), serveis amb credencials per defecte.
S3 — Capçaleres de seguretat HTTP
›Strict-Transport-Security (HSTS): força el navegador a usar HTTPS per a aquest domini durant un temps determinat.
›X-Content-Type-Options: nosniff: impedeix que el navegador "endevini" el tipus de contingut, mitigant alguns atacs XSS.
›X-Frame-Options / Content-Security-Policy (CSP): controlen si la pàgina es pot incrustar en un iframe (mitiga clickjacking) i quines fonts de scripts/estils es permeten.
›Server i X-Powered-By: revelen la versió exacta del software. Bona pràctica: eliminar-les o ofuscar-les (no donar pistes a l'atacant).
S4 — WAF (Web Application Firewall) bàsic
›Un WAF inspecciona el trànsit HTTP/HTTPS i bloqueja patrons coneguts d'atac (signatures d'injeccions SQL, XSS, etc.) abans que arribin a l'aplicació.
›ModSecurity: WAF de codi lliure, mòdul per a Apache i Nginx. S'utilitza amb un conjunt de regles (CRS - Core Rule Set d'OWASP) que cobreix la majoria de l'OWASP Top 10.
›Mode DetectionOnly vs Blocking: en producció es recomana començar en mode detecció (registra però no bloqueja) per ajustar regles i evitar falsos positius, abans de passar a mode bloqueig.

📋 Software i recursos addicionals
🔒
Certificats SSL per a serveis interns
Com que netcorp.local no és un domini públic, Let's Encrypt no pot emetre certificats per a la intranet. Cal generar certificats autosignats o muntar una CA pròpia amb OpenSSL.
Gratuït — OpenSSL inclòs a Ubuntu
Dues opcions
A.Per a la web pública (publicada a Internet amb domini real): Let's Encrypt + Certbot — gratuït i automàtic, requereix un domini real registrat.
B.Per a la intranet (intranet.netcorp.local): certificat autosignat amb OpenSSL, o crear una mini-CA i instal·lar-la com a "Autoritat de confiança" als PCs del domini via GPO (reaprofitant el P2).
🌍
Domini real per a la web pública (opcional)
Per provar Let's Encrypt amb un domini real, alguns registradors ofereixen subdominis gratuïts (DuckDNS, No-IP) que apunten a la IP pública del router de l'aula.
Gratuït — DuckDNS Opcional, no obligatori
Passos (si s'usa)
1.Crear un subdomini a duckdns.org (ex: netcorp-g1.duckdns.org)
2.Configurar-lo perquè apunti a la IP pública del router de l'aula
3.Verificar que el port forwarding (NAT estàtic, T2.3) fa arribar el trànsit al servidor web intern
